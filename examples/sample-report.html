skill-guide 使用说明书

把陌生 skill 解读成自包含 HTML 说明书的纯指导型 skill — 自己解读自己

▣skill-guide

1. 一句话结论

结论
已安装、可直接试用。把任意 skill 目录路径丢给它,会输出一份结构化 HTML 说明书 + 聊天摘要。解读阶段只读、零破坏。
当前判断
可以使用。SKILL.md 工作流写得极细,模板自包含、不依赖 CDN,触发语义覆盖了"这是什么 / 怎么用 / 完整度 / 搭配 / 修复 / 安装"等典型问法。
风险提示
输出一致性依赖模型自律,无 schema 校验。SKILL.md 用了大量"必须 / 不要"约束,但同一个 skill 跑两次,标色策略和文件关系块的具体形态仍可能略有差异,对外交付前要人工把关。

完整度
88 / 100
触发、流程、约束、模板、修复策略一应俱全。仅有缺渲染校验和 examples 单一两个非阻塞缺口。

直接试用
先丢一个真实 skill 目录给它跑一次。看看 HTML 报告的"快速判断"是否标色到位、文件关系块是否生成。
最大风险
输出风格依赖 LLM 自律
没有 schema 校验、没有渲染验证脚本。SKILL.md 用大量"必须 / 不要"约束,但生成结果一致性仍取决于模型。
直接使用风险
低
解读阶段只读,只生成单个 HTML 文件到当前工作区,不动原 skill,不调用付费 API,不需要登录态。

3. 直接试用

第一次试用 prompt
用 skill-guide 解读这个 skill,生成 HTML 说明书:

/Users/liuyun.1235/AI/codex/开放讨论/某个-skill

要求:
- 给完整度评分和推荐动作
- 列出搭配建议(标明已有 / 需安装)
- 给一段第一次试用的 prompt
- HTML 报告保存在当前工作区,聊天里只给摘要和路径
高阶用法
不用一开始就准备,想让结果更稳时再补充
Pro 版 — 让结果更准
补一句:"这个 skill 我打算搭配 frontend-design 使用",或"这是给设计师看的,模板要克制"。skill-guide 会把目标受众和搭配判断都写进报告。
Plus 版 — 复杂场景
· 多 skill 仓库:丢仓库根目录,先要"清单 + 工作流地图 + 推荐深读 1-3 个",再单独再来一轮深读。
· 失败案例对照:补一个你之前手写的说明 + 当时的痛点,让 skill-guide 知道你想避免哪类输出。
· 视觉升级:加一句"模板参考 image2 / 视觉母版",但要先给视觉方向图,不要让它把图片当背景。
· 自定义模板:把 assets/report-template.html 复制改造后,告诉 skill-guide 用你的模板,占位符按 SKILL.md 列表对齐。

4. 这个 skill 是什么

类型 — 单 skill(指导型)
目录顶层有 SKILL.md,可直接安装。没有可执行脚本,所有"产出能力"靠 LLM 按 SKILL.md 的约束生成 HTML — 这种 skill 没脚本一样可以是高完整度的。
包含什么(3 个文件)
SKILL.md
主流程 + 全部约束
assets/report-template.html
最终交付物模板
agents/openai.yaml
Codex/OpenAI 端 UI 元数据
怎么配合:SKILL.md 是入口,定义了"读什么 → 评什么 → 输出什么"的全流程,并列出 19 个占位符规范;report-template.html 是 SKILL.md 引导生成的最终容器,自带 CSS、复制按钮 JS、移动端响应式;openai.yaml 只在 Codex/OpenAI 端用来渲染卡片(brand_color、display_name),Claude Code 端不读它,但保留它对跨平台分发无害。
不负责什么
· 不安装 skill — 那是手动 ln -s 或包管理器的事。
· 不修改原 skill — 解读阶段强制只读,修复要再次确认。
· 不验证 skill 真的能跑 — 它评的是"看起来完整度",真正的功能验证要靠 verify 这类 skill 或人工试用。
· 不深读多 skill 仓库每一个 — 默认只做包级解读 + 推荐 1-3 个深读对象。

5. 搭配使用建议

单独用 — 够用
skill-guide 自己就能闭环:输入路径 → 输出 HTML 报告 + 聊天摘要。它的核心产物是判断和说明书,不依赖外部执行能力。
最佳搭配(后置)
· frontend-design 已有 — 当报告需要更高视觉规格(汇报、对外交付),让 frontend-design 在 skill-guide 输出后做 HTML/CSS 升级。
· verify 已有 — skill-guide 给出"试用 prompt"后,用 verify 真的跑起来确认行为,而不是只看报告判断。
可选搭配
· simplify / code-review 已有 — 如果用户从"解读"进入"修复副本"阶段,用这两个清理副本里的代码。
· relay-to-design-md 已有 — 反向场景:从 Relay 设计稿生成 design.md,再用 skill-guide 把它打包成 skill 的产出说明。
· deep-research 已有 — 当 skill 涉及陌生领域(某个行业的 PPT、某种数据格式),先做深度调研再让 skill-guide 解读会更准。

6. 什么时候适合用

最适合
拿到一个陌生的 skill / skill 包 / 多 skill 仓库,要快速判断"能不能用、值不值得安装、配多少时间深入"。也适合自己写完 skill 想生成对外说明书时。
什么时候用
· 朋友丢来一个 GitHub skill 链接、压缩包,或一个 SKILL.md 片段。
· 仓库里有 .claude/skills/ 或 skills/ 目录,要做盘点。
· 评审 / 选型 / 沉淀资料时,需要可分享的 HTML 说明书。
不适合
· 你已经熟悉这个 skill,只想快速调用 — 直接用就行,跑 skill-guide 是浪费。
· 想验证 skill 的真实行为 — skill-guide 不跑 skill,用 verify 或人工试用。
· 想批量修复 skill — 那是后续阶段,先解读再修。
· 输入是普通项目而不是 skill — 它会识别成 not a skill,产出价值有限。

7. 最好能产出什么

用户提供的资料示例形成的对应结果示例
一个单 skill 目录路径(SKILL.md + assets/) 完整 HTML 说明书,9 节齐全:结论、4 格快速判断、试用 prompt、文件关系图、搭配建议、边界、IO 表、风险、反馈单
一个多 skill 仓库根目录 包级 HTML:仓库总用途 + skill 清单表 + 工作流地图 + 推荐深读 1-3 个 skill 名单
一个 SKILL.md 片段(没 references / scripts) 识别为指导型 skill,完整度评分时不扣"缺脚本",而看触发、边界、流程是否清楚
一个普通项目目录(无 SKILL.md) 识别为 not a skill,说明为什么不是 skill,以及能否改造成 skill 的最小路径
一个 PPTX / DOCX 生成类 skill 报告里风险栏会强制提示"建议副本、预览、缩略图验证";搭配栏会推视觉、写作、图表能力
真实输出位置:HTML 文件保存在当前工作区(cwd),聊天回复给路径 + 30-80 字摘要;不写 stdout 长报告。真实输入接受:本地目录、SKILL.md 文件、压缩包、GitHub URL(需先下载到本地)。

8. 完整度与风险

完整度 88 / 100 — 接近可生产使用
SKILL.md 把"读什么 / 评什么 / 输出什么 / 不做什么 / 修复阶段怎么做"都写清楚了。频繁出现的"必须 / 不要 / 默认"让 LLM 自由发挥的空间被压到合理范围。
风险 1 — 缺渲染校验 一致性差异
用户会看到:没有 schema 或 HTML 校验脚本。同一个 skill 解读两次,占位符是否标色、文件关系块是横排还是堆叠,可能因模型采样而异。
影响:对外交付场景下,第一稿和第二稿可能不完全一致,需要人工把关。
怎么处理:关键交付物生成后用浏览器实际打开看一遍。后续可以加一个最小校验脚本,验证 9 个章节都在、`judge-grid` 有 4 格、关系块至少 3 个 step。修复咨询 prompt:
"在 assets/ 里加一个 validate-report.mjs,用 cheerio / jsdom 检查 sample-report.html 是否包含 9 个 section 标题、judge-grid 至少 4 个 .judge-value、relation-flow 至少 3 个 .relation-step。"
风险 2 — 示例单一 引导缺口
用户会看到:`examples/` 目前只有这份"自我解读"。多 skill 仓库的清单表 + 工作流地图、`not a skill` 项目的产物、PPTX 这类高风险 skill 的副本提示,都没有可参考的实物样本。
影响:第一次用的人不知道这些场景的产物长什么样,可能误以为 skill-guide 不支持。
怎么处理:用 skill-guide 跑一遍 frontend-design / pptx 这类典型 skill,把生成的 HTML 收进 `examples/`。每多一份示例,新用户理解成本下降一截。
风险 3 — 解读 / 修复边界靠用户表态 流程依赖
用户会看到:SKILL.md 强调"解读阶段只读、不默认修复",但当用户说"顺手把它改了"这类含糊指令时,LLM 是否进入修复阶段、是否做副本,取决于 LLM 对边界的判断。
影响:极少数情况下可能直接改原 skill 而非副本,尤其在多 skill 仓库扫描时。
怎么处理:用户侧:要修就明说"做修复副本,不要动原 skill"。skill 侧:可以在 SKILL.md "修复策略"一节加一句"任何修改前必须先确认副本路径并复述给用户"。

9. 试用失败后怎么找我修

建议动作顺序
1. 先试用 — skill 已安装为软链接(~/.claude/skills/skill-guide → /Users/liuyun.1235/AI/codex/开放讨论/skill-guide),触发即用,源目录改动立即生效。
2. 反馈样例 — 跑完一次后,把生成的 HTML、你期望但没出现的内容、看着不对劲的地方一起带回。
3. 修复副本 — 不直接改 开放讨论/skill-guide;复制一份到 开放讨论/skill-guide-fix-001/ 改,验收后再合回。
4. 归档 — 如果决定不用,把软链接删掉(rm ~/.claude/skills/skill-guide),源目录保留。
