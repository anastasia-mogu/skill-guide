frontend-design 使用说明书

Anthropic 官方放出的纯指导型 skill — 让 Claude 写前端时跳出"通用 AI 美学"

▣skill-guide

1. 一句话结论

结论
已安装、可直接试用。让 Claude 写网页 / 组件 / 海报 / dashboard 时,产出会更有"设计意图",远离 Inter + 紫色渐变这种 AI 套娃美学。解读阶段完全只读,零破坏。
当前判断
可以使用。SKILL.md 只有 42 行,但触发语义清晰、约束明确(列出禁用字体 / 禁用配色 / 必须做的事),是 Anthropic 官方 skill 的"低字数高密度"风格。
风险提示
效果强度上限取决于 prompt。一句"做个 landing page"和"做个 brutalist editorial 风格、1950s magazine 排版的 landing page",产出差距巨大。skill 自己不会替你做风格选择。

2. 快速判断

完整度
85 / 100
触发覆盖广(组件 / 页面 / dashboard / 海报)、禁用清单具体、原则简洁。扣分在没有正反例对照、没有 examples/、没有可复制的风格档案。
推荐动作
直接试用
写一个真实需求(landing page / dashboard / 海报)给它,先体感一下"AI 通用美学"和"frontend-design 加持"的差异。
最大风险
没有视觉锚点
SKILL.md 列了 11 种"风格关键词"(brutalist / 编辑型 / art deco……),但没附参考图、没附配色档、没附字体推荐。LLM 对每种风格的理解可能漂移。
直接使用风险
极低
纯指导型,不动文件、不调 API、不需要登录,只影响 Claude 写代码时的风格偏向。

3. 直接试用

第一次试用 prompt(基础)
用 frontend-design 给我做一个 landing page,主题:一个独立咖啡品牌叫 "MONO".

要求:
- 选一个"非通用"的设计方向(brutalist / editorial / art deco / 任何不常见的)
- 字体不要 Inter / Roboto / 系统字体
- 配色不要紫色渐变 + 白底
- 一定要有一个让人记住的视觉锚点(不规则版式 / grain 噪点 / 异形 mask 等)
- 输出可直接打开的 HTML/CSS,不依赖远程 CDN
高阶用法
不用一开始就准备,想让结果更稳时再补充
Pro 版 — 让风格落地更准
补一句风格锚定:"参考 1970s Swiss design / Wim Crouwel 字体网格"或"参考日本杂志 IDEA 的版式语言"。skill 知道你想要什么具体的设计语境,产出会显著锐化。
Plus 版 — 复杂场景
· 多页面体系:先让 frontend-design 输出一份"风格档案"(色板 / 字体对 / 间距尺度 / 装饰元素清单),后续每个页面都让 Claude 引用这份档案,避免风格漂移。
· 设计稿对照:你已有 Figma / Relay / 截图,把图丢给 Claude 一起 prompt,frontend-design 会偏向"匹配你的视觉",而不是它自己的偏好。
· React / Vue 工程:加一句"用 Motion 库做入场动画,只在 hero 区做一次 staggered reveal",避免 micro-interaction 过载。
· Dark mode 切换:明确说"做暗色主题为主,可选亮色",否则它会默认生成亮色。

4. 这个 skill 是什么

类型 — 单 skill(纯指导型,Anthropic 官方)
目录顶层只有 SKILL.md 和 LICENSE.txt,无脚本、无 references、无 examples、无 agents/openai.yaml。这是典型的"低字数高密度"指导型 skill — 整份说明 42 行,靠精确的禁用 / 必须项来约束 LLM 的风格输出。
包含什么(2 个文件)
SKILL.md
触发 + 全部设计原则
LICENSE.txt
Apache 2.0 授权
(无 examples / scripts)
所有效果靠 LLM 实时产出
怎么配合:SKILL.md 单文件承载全部能力,分四节:Design Thinking(让 Claude 先选风格再写)、Frontend Aesthetics Guidelines(字体 / 配色 / 动效 / 版式 / 背景五大维度)、禁用清单(Inter / Roboto / Arial / 紫色渐变 / 系统字体 / 通用版式)、复杂度匹配原则。LICENSE.txt 只是合规文件,不影响使用。
不负责什么
· 不替你选风格 — 你必须给主题或风格关键词,否则它只能默认猜。
· 不生成图片资源 — 配图 / 插画 / 图标要另外搭配。
· 不验证渲染效果 — 输出的 HTML 是否好看,要自己打开看,或搭配浏览器预览能力。
· 不做内容文案 — 它管的是视觉,文案需要你自己给或搭配写作型 skill。
· 不保证可访问性 — Maximalist 风格往往 a11y 较差,要主动要求"WCAG AA / 可键盘导航"。

5. 搭配使用建议

单独用 — 看场景
做单页 demo / 海报 / 一次性页面 — 单独够用。做完整产品 / 多页站点 / 长期维护的设计系统 — 不够,需要前置和后置搭配。
最佳搭配(前置)
· relay-to-design-md 已有 — 设计师已有 Relay / Figma 稿,先生成 design.md,再让 frontend-design 按 design.md 落地代码,避免它自由发挥跑偏。
· deep-research 已有 — 做陌生行业 / 陌生品牌时,先调研视觉参考,再让 frontend-design 锚定到具体的设计语言。
最佳搭配(后置)
· verify 已有 — 生成 HTML 后用 verify 真的打开页面,看渲染、看响应式、看动效是否过载。
· code-review / simplify 已有 — Maximalist 输出的 CSS 容易冗余,事后清理一遍。
可选搭配
· skill-guide 已有 — 做完一份成品后,反向用 skill-guide 把"我喜欢的风格选择 / 用了哪些字体 / 配色"沉淀成自己的 design profile。
· frontend-design 自身组合 — 多页面项目里,先让它生成"风格档案",后续每页都引用,自我搭配。

6. 什么时候适合用

最适合
需要一份"看上去不像 ChatGPT 默认风格"的前端产物 — landing page、品牌官网、Notion 替代品、海报、艺术家个人站、独立产品发布页、editorial 文章页等。
什么时候用
· 给客户做提案、做对外发布的页面、做有品牌调性的产品。
· 上一版 Claude 给的页面"看起来很 AI",想换风格。
· 想体验一下 brutalist / editorial / art deco 这些不常见美学的实现。
· 写 React / Vue 组件库,想要每个组件都有视觉锚点而不是"通用样式"。
不适合
· 内部 admin / 工具类页面 — 这些场景要克制和功能性,frontend-design 容易过度装饰。明确要求"极简、专业、无装饰"反而更好。
· 已有强约束的设计系统 — 公司有完整 design tokens / 组件库时,frontend-design 的风格选择会和现有系统冲突,改用 design 系统的具体规则。
· 对可访问性 / 性能要求极高的场景 — Maximalist 输出可能 a11y 差、首屏慢,需要明确指标约束。
· 后端 / 工具脚本 / 非视觉任务 — 完全无关。

7. 最好能产出什么

用户提供的资料示例形成的对应结果示例
"做个 landing page,主题是独立咖啡品牌 MONO" 一份完整 HTML/CSS,有明确风格(比如 brutalist 黑白 + serif 大标题 + grain 噪点),独特字体对(显示字体 + 正文字体),自定义动效(staggered reveal),可直接打开
"做一个数据看板,展示销售指标,要冷峻" 暗色主题 + 等宽字体 + 不规则网格 + 冷色高对比配色;数据卡片有微动画但克制;不会出现紫色渐变和默认 chart.js 配色
"给我做一张海报" 非常规版式(对角线 / overlap / 裁切),独特字体处理(变形 / 网格化 / 反白),装饰元素(几何 / 噪点 / 异形 mask),CSS-only 实现,可直接 print
"React 组件:一个文章卡片" 不止是 box-shadow + border-radius;会有 hover 时的揭示动画、独特的标签处理、有意识的字体层级,Motion 库加微交互(如果可用)
"美化这个已有页面"(给一段普通 HTML) 识别现有结构 → 选定一个统一风格方向 → 重写 CSS / 加装饰层 / 替换字体配色 → 给一两个标志性细节(custom cursor / scroll trigger / grain layer)
真实输出位置:HTML/CSS 直接写到 Claude 的回复里,或写到当前工作区的 .html 文件。真实输入接受:纯文字需求、设计图片(可读)、Relay/Figma 链接(配合其他 skill)、已有 HTML 片段。

8. 完整度与风险

完整度 85 / 100 — 可生产使用
42 行 SKILL.md 写得很精,触发宽、禁用具体、原则简洁。Anthropic 官方风格,低字数高密度。扣分主要在引导素材而非核心机制。
风险 1 — 缺少视觉锚点 理解漂移
用户会看到:SKILL.md 列了 brutalist / editorial / art deco / industrial 等 11 种风格关键词,但没有任何参考图、配色档、字体清单。LLM 对"brutalist"的理解可能是 90s 极简,也可能是粗野水泥纹理。
影响:同一句 prompt 跑两次,风格走向会有明显差异。需要靠 prompt 自己锚定参考来源。
怎么处理:用 Pro 版 prompt 补充参考(具体设计师 / 杂志 / 网站名)。skill 侧的修复咨询 prompt:
"在 frontend-design 旁加一份 references/style-archetypes.md,每种风格关键词配一行视觉参考(具体设计师名 / 网站名 / 字体推荐 / 经典色)。"
风险 2 — 没有 examples/ 看产物 引导缺口
用户会看到:没有示例 HTML / 截图,新用户不知道"frontend-design 启用前 vs 启用后"差距在哪。
影响:低估 skill 的价值,或对它的产物有不切实际的期待。
怎么处理:跑一次基础 prompt 留个产物在本地;团队里建 1 份"frontend-design 风格档案"作为内部参考。
风险 3 — 容易过度装饰 场景错配
用户会看到:对 admin 页面、工具类页面、纯内容型页面,LLM 会强行加视觉锚点和动效,结果反而干扰阅读 / 操作。
影响:给非品牌场景做工具页时,需要主动说"克制,不要装饰",否则容易过载。
怎么处理:明确说"refined minimalist,无动效,无装饰背景,只用 1 种字体"。SKILL.md 自己也提到了"minimalism 也是一种选择",但默认会偏向 maximalist。
风险 4 — 可访问性容易被忽略 默认弱项
用户会看到:SKILL.md 没强调 a11y。Maximalist 设计常见对比度不够、键盘焦点缺失、动效干扰阅读。
影响:对外发布前需要单独检查;不能直接交付给有 a11y 合规要求的客户。
怎么处理:prompt 里加"满足 WCAG AA 对比度,可键盘导航,prefers-reduced-motion 时禁用动效"。

9. 试用失败后怎么找我修

建议动作顺序
1. 先试用 — skill 已通过 cc-switch 单源 + 双端软链接安装(~/.claude/skills/frontend-design、~/.codex/skills/frontend-design 都指向 ~/.cc-switch/skills/frontend-design),触发即用。
2. 反馈样例 — 跑一次基础 prompt + 一次 Pro 版 prompt,把两份产物 + 你期望但没出现的视觉特征一起带回。
3. 修复副本 — 不直接改 cc-switch 里的 SKILL.md;复制到 ~/.cc-switch/skills/frontend-design-fix-001/ 改,验证后再合回。
4. 归档 — 不用就删两个软链接(rm ~/.claude/skills/frontend-design ~/.codex/skills/frontend-design),源目录保留。
反馈模板(可复制)
我跑了 frontend-design,prompt 是: [你的 prompt]
生成的 HTML: [path 或片段]

不对劲的地方:
1. [比如:还是用了 Inter / 系统字体]
2. [比如:版式很常规,没有视觉锚点]
3. [比如:配色就是紫色渐变 + 白底]

期望但没出现的:
- [比如:某个具体设计师的版式语言]
- [比如:特定的装饰元素(grain / 异形 mask / custom cursor)]

修哪里:
- 改 prompt(我给的不够具体)/ 改 SKILL.md 加视觉锚点 / 加一份 references 风格档案
还可以怎么用
· 自己有强烈视觉偏好时,不要让 frontend-design 自由选风格,先把参考图 / 设计师名 / 配色档喂给 Claude,再让 skill 落地。
· 团队里固定一份"我们的 frontend-design 风格档案"(色板 + 字体对 + 间距尺度),让所有页面引用,避免每个页面风格漂移。
· 学习用途:看 frontend-design 在不同 prompt 下选了什么字体 / 什么配色 / 什么版式语言,反过来扩充自己的设计词汇。
