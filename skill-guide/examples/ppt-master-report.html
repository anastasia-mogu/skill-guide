1. 一句话结论

结论
ppt-master 是一套真正生成可编辑 PowerPoint 的重型 PPT 工作流。它不是简单套模板,而是把资料转成 Markdown,确认设计规范,逐页生成 SVG,质检后再导出原生 PPTX 形状、文本框、图表和备注。
当前判断
值得试用,但不能当一键神器。仓库、脚本和示例都很完整,README 也明确说它是工具而不是许愿池。第一次建议用一份 5-8 页的小材料试跑,先熟悉确认节点和导出流程。
风险提示
最大风险是流程复杂和中途不可跳步。它要求串行执行、八项确认必须停下来等用户确认、SVG 逐页手写、质量检查后才能导出。随意跳步骤或批量脚本生成 SVG,会直接违反它自己的核心规则。

3. 直接开始用

第一次试用提示词
请用 ppt-master 试做一份小型 PPT,先不要跑大项目。

材料:
- 我会提供一份 5-8 页以内的 Markdown 或 PDF
- 目标是生成 6-8 页 16:9 PPT
- 风格先用自由设计,不要套模板

要求:
- 严格按 ppt-master 的串行流程执行
- 八项确认时必须先停下来让我确认
- 第一次不生成 AI 图片,图片用占位或材料内已有图片
- 导出前运行 SVG 质量检查
- 产物保存到当前工作区,聊天里只告诉我路径和需要人工检查的点
高阶用法
不用一开始就准备,想让结果更稳时再补充
Pro 版 — 降低返工
提前说明页数、受众、语气、是否保留原文、是否允许重构内容,可以让八项确认更少来回。
请在刚才的基础上补充这些约束:

- 受众:公司内部复盘会,听众不是技术人员
- 页数:8 页左右,宁可少而清楚
- 内容处理:可以合并、删减和重排,但所有关键事实必须来自源材料
- 风格:克制、专业、有图表层次,不要花哨动效
- 图片:优先使用材料内图片;没有就用占位,不要自动生图
- 导出后请列出 PowerPoint 里最应该人工检查的 3 个点
Plus 版 — 复杂场景
· 已有 PPT 模板:明确说是填充模板、保留页数美化,还是把旧 PPT 当资料重构。
· 有数据图表:导出前追加图表校准,避免像素坐标误差。
· 长材料:拆成 Phase A / Phase B,先确认设计规范和大纲,再换新窗口继续生成。
· 需要旁白:明确是否生成演讲者备注和音频旁白,不要默认启用。 
请按复杂场景处理这次 ppt-master 任务:

- 如果我给的是现有 PPT,先问清楚:保留原页数和原文做 beautify,还是把它当源材料重新组织
- 如果有图表页,导出前运行 verify-charts 工作流
- 如果源材料很长,请建议 split mode:先完成设计规范和资源准备,再让我在新窗口继续生成
- 如果需要图片,先把图片来源分成 user / web / ai / placeholder,不要擅自调用生图
- 如果需要旁白,先问我选择语音后端和声音,不要直接调用 notes_to_audio.py
失败了怎么反馈
带回项目目录、终端报错、当前停在哪一步、已生成的 design_spec.md / spec_lock.md / svg_output/ / exports/ 状态。最有用的反馈是:"卡在环境安装、八项确认、SVG 质检、导出 PPTX,还是 PowerPoint 打开效果不对"。

4. 和什么技能最搭配

单独用 — 能用,但不轻
ppt-master 自带资料转换、项目管理、确认页面、预览、质检、导出和旁白脚本,单独能跑完整链路。问题是链路长,第一次最好配合预览、代码审查和文件验证来降低返工。
最佳搭配(前置)
· 已有deep-research — 只有主题、没有材料时,先调研并形成可引用资料,再交给 ppt-master。
· 已有frontend-design — 对外汇报或品牌型 PPT,先明确视觉方向,再让 ppt-master 写设计规范。
最佳搭配(后置)
· 已有verify — 导出后检查 PPTX 是否存在、能否打开、预览图和页面数量是否符合预期。
· 已有code-review / simplify — 修改脚本或修复导出问题时再用;普通生成 PPT 不需要主动介入代码。
可选搭配
· 可选imagegen — 当项目确认要 AI 图片,并且本地 API 后端不可用时,可走人工/宿主生图路径。
· 可选skill-guide — 每次跑完一个案例后,反向总结失败点,沉淀自己的 PPT 生成 SOP。

5. 产出物对照

输入投入你提供什么通常能拿到（高亮＝多给后多得的）
最少 一个主题、一段文字,或一个小型 Markdown / PDF 项目目录、设计确认建议、初步设计规范、SVG 页面、演讲者备注和可编辑 PPTX;但内容事实、图像来源和风格选择会更依赖模型判断。

充分 ＋源 PPT 模板、品牌/版式模板目录、示例成品、运行环境、失败案例 前述全部,并 更接近可交付工程产物:可走模板填充、旧 PPT 美化、图表校准、对象级动画、音频旁白和重复导出;还能保留备份用于复导。
以上为通常可形成的预期(非承诺);给得越准,结果越准。真实输出通常在项目目录内,包括 design_spec.md、spec_lock.md、svg_output/、notes/、exports/*.pptx、backup/。

6. 完整度与改善点

完整度 91 / 100 — 接近生产级
主流程、脚本、模板、参考文档、示例、插件元数据、中文/英文 README 都很完整。扣分主要不是能力不足,而是流程过重、安装路径多、版本标识不完全一致,以及新手容易误以为它能一把出完美成品。
改善点 1 — 版本标识不一致 容易困惑
用户会看到:英文 README 标 v2.10.0,中文 README 标 v2.9.0,marketplace 元数据标 2.7.0。
影响:用户不知道自己看到的是不是最新版,安装问题也难对齐。
怎么处理:以 GitHub main 当前文件为准;如果要修,先统一 README 和 marketplace 的版本字段。
改善点 2 — 安装路径多 新手容易装半截
用户会看到:下载 ZIP、git clone、skill marketplace 都能装,但 marketplace 只拉 skill 文件,仍要在安装目录跑依赖。
影响:可能以为安装 skill 就能直接导出 PPTX,实际脚本依赖没装。
怎么处理:第一次建议完整 clone 仓库,在仓库根目录跑 pip install -r requirements.txt。
改善点 3 — 输入文件会被移动 需要先备份
用户会看到:Step 2 明确要求 import-sources --move,源文件会进入项目 sources/。
影响:如果用户只放了一份原件,会误以为文件丢了。
怎么处理:试跑前复制一份材料到项目目录,不要直接拿唯一原件测试。
风险点 1 — 真实运行会消耗模型/API 费用和时间不可忽略
用户会看到:README 明确说工具免费,主要成本是模型使用;最佳效果推荐 Claude 大上下文和 AI 生图。
影响:长材料、AI 图片、音频旁白会明显增加时间和费用。
怎么处理:第一次关掉 AI 图片和旁白,用小材料试跑;确认流程后再上大项目。

7. 适合谁用，不能做什么

它能帮你做什么
ppt-master 帮你把资料变成真正的 PowerPoint 文件,而不是一页一张图片。它适合把 PDF、Word、网页、Markdown、已有 PPT 或直接粘贴的内容,经过设计规范确认和逐页 SVG 生成后,导出成可以在 PowerPoint 里继续改的 PPTX。
文件关系图
包里有什么
SKILL.md645 行主工作流,规定串行步骤、确认点、输入输出和禁止事项。
scripts/资料转换、项目管理、图片处理、SVG 质检、PPTX 导出、旁白等执行脚本。
references/策略师、执行器、共享 SVG/PPT 技术约束、图片策略等角色说明。
workflows/模板填充、美化旧 PPT、实时预览、图表校准、音频旁白等旁路流程。
templates/版式、品牌、图表、图标库和设计规范参考模板。
examples/大量已生成案例,用于观察能力上限和回归验证。
主流程:从输入到 PPTX
用户给材料PDF / DOCX / URL / Markdown / PPTX / 文字
source_to_md + project_manager转换成 Markdown,创建项目,移动源文件到 sources/
Strategist八项确认后写 design_spec.md 和 spec_lock.md
Executor + live preview逐页手写 SVG,启动本地预览,运行质量检查
finalize_svg + svg_to_pptx后处理并导出原生可编辑 PPTX
旁路文件:不直接生成主 PPT,但帮助扩展和分发
README / docs/安装、FAQ、路线图、Windows 指南和解释材料
.claude-plugin/Claude Code plugin marketplace 元数据
LICENSEMIT 授权,影响分发合规
一句话看懂:真正跑流程的是 SKILL.md + scripts + references + templates;README 和 marketplace 负责安装/分发;examples 用来帮你判断成品上限。
什么时候最适合
· 你要的是可继续编辑的 PowerPoint,不是网页演示或整页图片。
· 你有 PDF、Word、网页、Markdown 或已有 PPT,希望重构成一份正式汇报。
· 你愿意先确认设计规范,再让 AI 做大部分生成,最后自己在 PowerPoint 里打磨。
· 你需要模板填充、旧 PPT 美化、图表页、备注甚至音频旁白这类复杂链路。
不能做什么
· 不能保证一把出完美成品;README 自己也强调还需要你打磨。
· 不能跳过八项确认和串行流程;确认点是它的核心安全阀。
· 不能在没有 Python 依赖的环境里直接导出 PPTX。
· 不能替代 PowerPoint 审稿;导出后仍要人工检查字体、图表、动画、备注和播放效果。
· 不适合只想快速生成一张海报、一个网页或纯文本摘要的轻任务。
