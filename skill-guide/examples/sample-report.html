1. 一句话结论

结论
skill-guide 是 skill 说明书生成器。把一个 skill 路径、SKILL.md 或多 skill 仓库交给它,它会读结构、判完整度、给试用提示词,并生成一份可预览、可存档的 HTML 报告。
当前判断
可以直接使用。最新版默认 6 节,发现真实可打开 example 时才插入"产出物示例",并补了快速判断、Pro/Plus 复制区、文件关系图、搭配标签、风险命名和交付前最小自检。
风险提示
最大缺口已从结构校验转向内容判断。现在可以用脚本检查 6/7 节、5 个快速判断格、复制块、搭配标签、星级和文件关系图;但内容准不准、风险排序对不对、版式舒不舒服仍要人工看。

3. 直接开始用

第一次试用提示词
用 skill-guide 解读这个 skill,生成 HTML 说明书:

./skill-guide

要求:
- 给完整度评分和推荐动作
- 列出搭配建议(标签标明本机安装状态,星级放在分组标题里)
- 给一段第一次试用提示词
- 如果仓库里有可打开的真实 example,第 5 节展示产出物示例;没有就不显示第 5 节
- 生成 HTML 后按交付前最小自检检查 6/7 节、5 格、copy-block、badge、文件关系图和旧章节残留
- HTML 报告保存在当前工作区,聊天里只给摘要和路径
高阶用法
不用一开始就准备,想让结果更稳时再补充
Pro 版 — 让结果更准
补一句报告对象和使用场景,让 skill-guide 少讲内部分类,多讲用户能拿它做什么。适合做分享版、验收版或给非技术用户看的说明书。
请在刚才的基础要求上继续补充:

- 这份报告是给非技术用户看的,少讲内部分类,多讲怎么开始用
- 请保持 6/7 节结构,重点检查快速判断、文件关系图、搭配标签和真实示例展示
- 如果发现高风险问题才写"风险点",普通缺口写"改善点"
- 交付前请按最小自检做一遍结构复核
Plus 版 — 复杂场景
· 多 skill 仓库:丢仓库根目录,先要"清单 + 工作流地图 + 推荐深读 1-3 个",不要一次性深读全部。
· 稳定性实验:让它用同一个 skill 连续生成两版,对比 6/7 节结构、快速判断和高阶提示词是否稳定。
· 失败案例对照:补一个你不满意的旧报告,让 skill-guide 判断该修 SKILL.md、模板还是提示词。
· 模板验证:要求生成后用浏览器检查 631px 和 390px 两档,确认没有横向溢出。 
请按复杂场景处理这次解读:

- 如果这是多 skill 仓库,先做包级清单和工作流地图,不要一次性深读全部
- 如果有旧版报告或失败案例,请对照指出问题来自内容规则、模板样式还是提示词
- 生成 HTML 后按"交付前最小自检"检查 6/7 节标题、5 个快速判断格、Pro/Plus 复制块、badge 标签、文件关系图和移动端不溢出
- 最终报告仍然保存为自包含 HTML
失败了怎么反馈
带回输入路径、生成的 HTML、你觉得不对的地方、期望但没出现的内容。要修规则就说"改 SKILL.md",要修版式就说"改模板",要修成品就说"只改这个 HTML"。

4. 和什么技能最搭配

单独用够不够
只要目标是"看懂一个 skill、判断能不能试、生成说明书",skill-guide 自己就能闭环:输入路径 → 读取结构 → 输出 HTML 报告 + 聊天摘要。
最值得搭配 ★★★★★
· 安装状态未知verify — skill-guide 给出第一次试用提示词后,用 verify 真的跑起来确认行为,而不是只看报告判断。
· 安装状态未知frontend-design — 当报告要对外交付或展示,在 skill-guide 输出后优化 HTML/CSS 的视觉细节。
按需搭配 ★★★
· 安装状态未知code-review / simplify — 从解读进入修复副本阶段后,用它们清理代码、压缩冗余和查回归。
· 安装状态未知browser — 成品 HTML 需要验响应式、复制按钮、横向溢出时,用浏览器做最终检查。

6. 完整度与改善点

完整度 96 / 100 — 可以直接使用
SKILL.md 已经把"读什么 / 评什么 / 输出什么 / 什么时候停 / 如何命名风险 / 如何保持版式稳定"写清楚了,并新增结构校验脚本。模板也同步了 6/7 节目录、Pro/Plus 复制区、搭配标签、星级和移动端适配。
改善点 1 — 内容判断还不能脚本化 人工判断
用户会看到:脚本能发现结构漂移,但不能判断"最大风险"是否真的最大,也不能判断文案是否容易懂。
影响:报告会更稳定,但最终仍需要人看一遍内容和版式。
怎么处理:生成后先跑 python3 skill-guide/scripts/validate-report.py,再用浏览器预览重点看内容判断和窄屏观感。
改善点 2 — 多做临时实验 泛化验证不足
用户会看到:examples/ 只保留自我解读和 frontend-design 两份核心样例,没有把多 skill 仓库、文件产物型、非 skill 项目和文件编辑类高风险 skill 都长期塞进仓库。
影响:新规则在复杂场景里是否稳定,还缺真实对照。
怎么处理:接下来拿一个多 skill 仓库或文件类 skill 做临时实验,只在规则有价值时反推 SKILL.md 约束,不把所有实验报告长期收进 examples。
改善点 3 — 修复副本规则还能再硬一点 边界提醒
用户会看到:SKILL.md 已写"解读阶段只读、不默认修复",但修复阶段只说优先创建副本,还没有要求"动手前复述副本路径"。
影响:用户一旦用含糊说法要求修改,模型可能需要临场判断是否直接改原 skill。
怎么处理:在"修复策略"里补一句:任何修改前必须确认副本路径、列出会改哪些文件,再行动。这个问题目前不阻塞第一次试用,所以先标改善点,不是风险点。

7. 适合谁用，不能做什么

它能帮你做什么
skill-guide 能把陌生 skill 解读成一份可保存、可分享的 HTML 说明书。它回答"这个东西能不能用、第一次怎么试、该搭配谁、哪里还不稳",让非技术用户也能快速判断值不值得继续投入。
文件关系图
包里有什么
SKILL.md核心说明书,决定这个 skill 怎么被触发、怎么读材料、怎么生成报告。
assets/放报告模板和展示资源,目前主要是自包含 HTML 模板。
examples/放生成后的示例报告,用来对照内容和版式是否稳定。
agents/放 Codex / OpenAI 端展示元数据,不参与报告正文生成。
README.md给人看的安装、触发、示例和已知缺口说明。
LICENSE仓库根层授权文件,不在 skill 子目录里。
主流程:从输入到报告
用户给 skill 路径目录、SKILL.md、压缩包或仓库根目录
SKILL.md决定读什么、怎么判断、报告写成默认 6 节还是带示例的 7 节
assets/report-template.html把判断内容装进可预览 HTML,负责目录、卡片、表格和复制按钮
examples/*.html作为成品样例和回归对照,观察规则改完后是否稳定
旁路文件:不直接生成报告,但帮助使用和分发
README.md给人看的安装、触发、示例和已知缺口说明
agents/openai.yamlCodex / OpenAI 端展示元数据,不参与报告生成
LICENSE在仓库根层负责授权,skill 子目录没有单独许可证文件
一句话看懂:真正生成报告的是 SKILL.md 和模板;示例负责验证稳定性;README、OpenAI 元数据和 LICENSE 是使用、展示、分发层面的旁路文件。
什么时候最适合
· 拿到一个陌生 skill,想快速判断值不值得试。
· 想把自己写的 skill 变成对外说明书或交付说明。
· 要盘点多 skill 仓库,先做清单、关系图和推荐深读对象。
· 想比较同一个 skill 多次生成报告时,内容和版式是否稳定。
不能做什么
· 不替你安装 skill — 安装仍要用软链接、包管理器或手动复制。
· 不默认修改原 skill — 解读阶段只读,修复要再次确认。
· 不证明被解读 skill 一定能跑 — 它先评文档、结构和风险,真正执行验证要靠 verify 或人工试用。
· 不深读多 skill 仓库每一个 — 默认只做包级解读 + 推荐 1-3 个深读对象。

由 skill-guide 按最新版规则自我解读生成 · 示例报告 — 用于观察内容和版式稳定性
