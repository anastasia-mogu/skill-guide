# Skill Guide

把陌生 skill 解读成用户能直接使用的 HTML 说明书。不要只总结 `SKILL.md`，要判断它是什么、能做什么、怎么和它对话、需要准备什么、真实输入输出是什么、完整度如何、适合搭配什么能力、第一次怎么安全试用，以及试用后如何反馈修复。

默认判断：用户还处在评估阶段。除非用户明确要求，不要安装、修复、合并、改写原 skill，或执行可能修改真实文件/线上状态的动作。

## 核心原则

- 先读再问。先检查目录结构、`SKILL.md`、`references/`、`scripts/`、`assets/`、`agents/openai.yaml`、明显入口文件和示例，再向用户补问。
- 先识别对象类型。区分单 skill、skill 包、多 skill 仓库、普通项目内嵌 skill、非 skill 项目。
- 解读阶段保持只读。发现问题先记录，不默认修复。
- 区分“用户友好用法”和“真实执行输入”。如果文档说“给主题”，但脚本实际要 JSON、图片、API key 或本地服务，要补出中间步骤。
- 默认给出搭配建议。判断它单独是否够用，哪些能力放在前置/后置会更好；只有发现相似 skill 时才展示合并建议。
- 高风险文件类 skill 要建议副本、预览和验证，例如 PPTX、DOCX、PDF、设计文件、代码迁移、批量改图。
- 最终产出默认生成信息优先的自包含 HTML 报告。聊天里只给精简摘要和 HTML 路径；只有用户明确说不要 HTML 时，才只输出对话版。

## 对象识别

先给出对象类型：

- `single skill`：一个目录顶层有 `SKILL.md`，通常可直接安装。
- `skill package`：一个目录或压缩包里包含一个或多个可安装 skill。
- `multi-skill repository`：项目里有 `.claude/skills/`、`.codex/skills/`、`skills/`、`agent-skills/` 等多个 skill。
- `project with embedded skills`：普通项目里把 skill 作为辅助材料。
- `not a skill`：没有 `SKILL.md` 或 skill 元数据；说明它为什么不是 skill，以及能否改造成 skill。

多 skill 仓库不要默认深读每一个。先做包级解读、skill 清单、关系图和推荐深读顺序；只有在用户指定或最有价值的 1-3 个对象上继续深读。

## 检查清单

检查可见的内容：

- Frontmatter：`name`、`description`、触发语义、是否覆盖用户真实问法。
- 目录结构：`SKILL.md`、`references/`、`scripts/`、`assets/`、`agents/openai.yaml`。
- 工作流：步骤是否完整、有序、安全，是否说明何时停止或询问用户。
- 真实输入输出：文件、图片、URL、API key、环境变量、本地服务、浏览器状态、设计工具、模板、生成路径。
- 一致性：文档路径、脚本名、模板路径、命令示例、模型名、依赖说明是否和真实文件一致。
- 依赖：Python/Node 包、系统工具、网络、MCP、浏览器、登录态、外部应用。
- 风险：是否会覆盖原文件、批量修改、调用付费 API、连接线上服务、缺少 dry-run。
- 验证：测试、预览图、截图、缩略图、schema 校验、lint/build、输出检查。

## 完整度评分

给出 0-100 分和一句判断：

- `90-100`：接近可生产使用，说明、边界、依赖、验证都清楚。
- `70-89`：可用，但有小的依赖、文档或验证缺口。
- `50-69`：可试用但脆弱，需要引导或修复副本。
- `30-49`：像半成品或 demo，只建议阅读，不建议依赖。
- `<30`：当前不可用。

不要只看代码数量。指导型 skill 没有脚本也可以很完整，只要边界、触发、流程和成功标准清楚。

## token 消耗预估

帮用户在决定用不用之前，先看到一次解读大概花多少 token。给一个本次预估值，并附粗略参考区间：

- 预估区间分三档，数字只是**粗略示意**，要标注“粗略”：
  - `轻 ~3-5k`：单 skill、指导型、无脚本，只读 `SKILL.md`。
  - `中 ~6-10k`：带 `scripts/`、`references/`，需读若干引用文件。
  - `重 ~12k+`：多 skill 仓库、深读多个对象，或生成较大报告。
- 影响因素：skill 大小、是否深读引用文件、是否多 skill 仓库、是否生成 HTML。
- 给 1-2 条省 token 建议，例如“先不深读引用文件，按需再补”“多 skill 仓库先做包级解读，不一次性深读整库”。

## 搭配与合并判断

每次都回答：

- 它能不能单独完成任务？
- 它缺的是内容策略、视觉设计、图片生成、数据准备、环境执行、文件验证、浏览器预览，还是领域知识？
- 哪些 skill/能力应该在它前面或后面？
- 现在应该串联使用，还是以后考虑合并？

默认建议“搭配”，不默认“合并”。只有当多次真实使用证明触发词高度重叠、输入输出不可分、职责长期重复时，才建议合并。没有发现相似 skill 时，不展示“合并建议”占位。

常见搭配：

- PPTX 生成类：搭配视觉设计、写作/叙事、图片生成、表格/图表、缩略图验证。
- 知识转网页类：搭配知识结构化、前端设计、浏览器预览、图片生成。
- 纯设计指导类：通常放在执行型 skill 前面，不轻易合并。
- 文件编辑类：搭配预览、渲染、diff、备份和验证能力。

## 输出结构

默认按这个顺序输出：

1. **一句话结论**
   - 直接回答能不能用、现在建议怎么做、最大风险是什么。
   - 非必要不夹英文。skill 名、文件名、命令、路径、专有模型/产品名可以保留英文；普通说明优先用中文，例如“网页”优先于 `landing page`，“仪表盘”优先于 `dashboard`。
   - 高亮不要放在“只读、零破坏”这类流程安全声明上，优先高亮“它是什么 / 能干什么 / 解决什么问题”。

2. **快速判断**
   - 默认 4 个判断：skill 完整度、推荐动作、最大风险、直接使用风险；再加一个“token 消耗预估”格（本轮新增），共 5 格。
   - 版式必须保持“速览”：前 4 格用双列卡片，token 消耗预估单独占一整行。不要把快速判断写成大段正文。
   - 顶部数据条不是必需项；如果展示，不能挤占快速判断的主卡片。
   - 每格必须先给一个短主判断，再给补充说明。主判断要单独标色：可用/能试/低风险用绿色，分数/重点数值用强调色，风险/缺失/阻塞用红色。
   - “完整度”和“推荐动作”两格只给主判断即可，不要加小字说明；让它们承担快速体感判断。
   - 快速判断不能整段都用普通正文颜色；如果报告里看不出“哪个值最重要”，说明这一格生成失败，需要重写。
   - 去掉不好理解的小字说明；解释性内容放到后面的风险或边界模块。
   - 不默认展示对象类型、当前状态、输入形态这类内部字段，除非它们会影响下一步动作。

3. **直接开始用**
   - 给一段可直接复制的第一次试用提示词。
   - 如果有多个典型场景，用多个示例卡纵向展示。
   - 准备清单不单独成章，放在这里作为“想让结果更准时补充什么”。
   - 补充信息用“高阶用法”卡片展示，样式和示例卡一致。
   - 高阶用法分两档：`Pro 版` 放最小增强信息；`Plus 版` 放失败案例、依赖、模板、脚本、参考文档、资源等进阶信息。
   - 高阶用法如果给了可直接使用的补充提示词，也要提供复制按钮；不要只给一段不可复制的说明。
   - `Pro 版` 和 `Plus 版` 的复制按钮要放进对应的白色提示词框右下角，和真正要复制的内容放在一起，作为整段提示词的收尾操作。不要把按钮放在小标题同一行、提示词框右上角或整块高阶用法的头部。
   - 可复制提示词块统一用 `.copy-block` 包住：按钮在 `.copy-block` 内，目标 `pre` 紧跟其后。
   - 可复制内容必须是用户能直接发给 AI 的提示词，不要只复制说明文字。
   - 最下方放“失败了怎么反馈”：只保留最短反馈模板或反馈材料清单，告诉用户如果报告不对该带回什么材料、要改规则/模板/成品时怎么说。

4. **和什么技能最搭配**
   - 放在“直接开始用”后面，先回答这个 skill 单独够不够用，再告诉用户下一步该搭配谁。
   - 先给“单独用：够用/不够用/看场景”这类明确判断。
   - 再给“最佳搭配 / 后续搭配 / 可选搭配”。
   - 每个搭配对象都要把状态标签放在最前面，并使用 `.badge` 药丸样式。示例：`<span class="badge">已有</span><strong>skill-name</strong>`、`<span class="badge need">需安装</span><strong>skill-name</strong>`、`<span class="badge unknown">不确定</span><strong>skill-name</strong>`、`<span class="badge optional">可选</span><strong>skill-name</strong>`。
   - 只有发现相似 skill 时才展示“合并建议”，并说明建议和谁合并、为什么、下一步怎么做。

5. **产出物对照**
   - 核心是“用户不同程度的输入 → 对应的产出物”：主语是“用户给多少料”，不是“这是什么对象”，也不是“工具读多深”。
   - 用“输入投入”递进三档表，全部针对当前这一个被解读对象，任何 skill 都成立：
     - `最少`：只给一个 skill 路径 / SKILL.md → 基础报告，能判断能不能用，但真实 IO、依赖偏推测。
     - `补充`：再给依赖说明、真实输入样例、跑过的例子 → 真实 IO 坐实、依赖确认、风险/验证更准。
     - `充分`：再给失败案例、运行环境、已有产物样例 → 可直接交付：验证到位、能分享存档。
   - 表格三列：`输入投入`、`你提供什么`、`通常能拿到`。每档在“通常能拿到”里**高亮比上一档多给后多得的部分**，用 `<span class="diff">…</span>` 标出。
   - 与第 3 节“高阶用法”互补：第 3 节讲“补什么”（输入端清单），第 5 节讲“补了能得到什么”（输出端结果），不要写重。
   - 措辞统一“通常可形成”，不能写成承诺、不能写成“已生成的真实文件”。解读阶段只读，禁止为了填表去执行被解读 skill 生成产物。
   - 可选附注：若被解读 skill **自带**示例产物文件（其 `examples/`、`assets/` 成品），可在“充分”档用相对路径附“skill 自带示例”可点击链接作为佐证；没有就纯文字，不虚构链接、不内嵌外链 CDN 图片。
   - tfoot 收一句注脚：“以上为通常可形成的预期（非承诺）；给得越准，结果越准。”

6. **完整度与改善点**
   - 分数、缺失项、依赖问题、路径不一致、潜在破坏性、验证缺口。
   - 默认写“改善点”，只有问题会阻塞试用、造成破坏、导致明显误判或真实风险时，才标成“风险点”。
   - 如果风险足够高，标题必须明确写“风险点”，不要为了语气温和而淡化。
   - 改善点要写清楚“用户会看到什么 / 对用户有什么影响 / 怎么处理”，必要时提供可复制修复咨询提示词。

7. **适合谁用，不能做什么**
   - 合并原“这个 skill 是什么 / 什么时候适合用 / 试用失败后怎么找我修”。这一节只负责说明用途、适用场景、边界和文件关系，避免在前面行动章节里重复。
   - 第一块写“它能帮你做什么”：用白话说明实际用途，不要先讲“单 skill / 纯指导型 / 包类型”这类分类名。分类只在影响安装或判断时顺带提一句。
   - 第二块写“什么时候最适合”：只列 2-4 个典型场景，不重复“它能帮你做什么”的定义。
   - 第三块写“不能做什么”：用用户能理解的话列边界，标题用“不能做什么”，不用“不负责什么”。
   - 如果识别到 2 个以上关键文件或目录，必须给“文件关系图”。关系图先写“包里有什么”，再写“主流程：从输入到报告”，最后写“旁路文件：不直接生成报告，但帮助使用和分发”。
   - “包里有什么”要解释 skill 包里的关键文件或文件夹分别代表什么，例如 `SKILL.md`、`assets/`、`examples/`、`scripts/`、`references/`、`agents/`、`README.md`、`LICENSE`。
   - “主流程”要解释真实使用中谁先被读取、谁形成最终产物；“旁路文件”只放 README、授权、展示元数据这类不直接参与报告生成的内容。
   - 文件关系图优先上下排列，便于自适应长文件名和移动端；不要用不适合长文字的横向挤压格子。
   - 路径和来源只在有助于用户判断时展示，避免技术字段挤占主线。

不要单独保留“对话方式”章节。对话建议要直接体现在可复制提示词和示例卡的小字说明里。

不要单独保留“修复与安装策略”和“试用后反馈单”章节。它们统一收进“直接开始用”的尾部反馈入口。

## 报告稳定性规则

为了让同一个 skill 多次解读时内容和版式稳定，以下规则优先级高于临场发挥：

- **章节稳定**：默认报告固定 7 节，章节名和顺序不要随意改。不要恢复旧章节，例如“什么时候适合用”“试用失败后怎么找我修”。需要补充时并入现有 7 节。
- **职责稳定**：第 3 节最后放失败反馈入口；第 4 节只讲搭配；第 5 节只讲产出物对照；第 6 节只讲完整度、改善点和风险点；第 7 节只讲用途、适用场景、边界和文件关系。
- **卡片节奏稳定**：卡片内部不要把标题、说明、可复制内容挤在一起。标题行、说明段、可复制提示词块要分开。
- **高阶用法稳定**：`Pro 版` 和 `Plus 版` 各自独立成 `.item`。标题只放在 `.item-head`；说明放 `.value`；真正可复制给 AI 的完整提示词放进 `.copy-block > pre`，复制按钮放在同一个 `.copy-block` 的右下角。复制内容不能只是说明文字。
- **快速判断稳定**：完整度、推荐动作只给主判断，不加小字；最大风险、直接使用风险、token 预估可以有简短说明。前四格双列，token 单独一整行。
- **样式稳定**：搭配状态必须用 `.badge` 药丸样式，不能退化成纯文本方括号。文件关系图用上下排列。风险足够高时标题写“风险点”，普通问题写“改善点”。
- **高亮稳定**：一句话结论里的高亮优先标“它是什么 / 能做什么 / 解决什么问题”，不要高亮流程安全声明。

## 交付前最小自检

生成 HTML 报告后，必须先做一次最小结构自检，再把路径交给用户。这个自检只负责防止报告结构和关键交互漂移，不替代人工判断内容好不好、版式美不美、浏览器渲染是否完全正确。

优先检查这些项目：

- 章节：目录和正文都必须是 7 节，顺序为“一句话结论 / 快速判断 / 直接开始用 / 和什么技能最搭配 / 产出物对照 / 完整度与改善点 / 适合谁用，不能做什么”。
- 旧章节名：不得残留“直接试用”“搭配使用建议”“什么时候适合用”“最好能产出什么”“试用失败后怎么找我修”等旧结构。
- 快速判断：前 4 格为双列判断，`token 消耗预估` 单独占一整行；“完整度”和“推荐动作”只给主判断，不加小字。
- 直接开始用：必须包含第一次试用提示词；`Pro 版` / `Plus 版` 若提供可复制内容，必须使用 `.copy-block`，复制按钮放在同一个白色提示词框右下角。
- 失败反馈：第 3 节底部必须有“失败了怎么反馈”，不要单独成章。
- 搭配标签：搭配对象状态必须用 `.badge` 药丸样式，并放在 skill 名前面。
- 文件关系图：第 7 节如涉及 2 个以上关键文件或目录，必须包含“包里有什么 / 主流程 / 旁路文件”。
- 占位符：不得残留 `{{...}}`、`TODO`、空标题、空卡片或明显模板文案。
- 移动端风险：长文件名、复制按钮、表格和关系图不能明显横向溢出；无法自动验证时至少用浏览器窄宽度快速看一眼。

如果仓库里已有校验脚本，先跑脚本；没有脚本时，用 `rg` / `sed` / 浏览器预览做人工最小检查。自检失败必须先修正 HTML，再交付路径。聊天里只汇报关键结果，不把整份检查清单贴给用户。

## 多 Skill 仓库输出

如果输入是多 skill 仓库，先输出：

- 仓库总体用途。
- Skill 清单表：名称、一句话用途、输入、输出、依赖、完整度、推荐动作。
- 工作流地图：上游、下游、旁路工具。
- 推荐优先深读的 1-3 个 skill。
- 只有当继续深读会改变范围、成本或产物方向时才询问用户；普通分层解读不要每一层都打断。

## 最终产出策略

默认最终产出是自包含 HTML 报告，而不是只在对话里输出长说明。优先把最重要的信息放在最容易被用户看到的位置：

- 先给结论：能不能用、完整度、是否阻塞第一次试用。
- 再给行动：第一次怎么试、用什么提示词、想更准时补什么信息。
- 再给搭配：适合和什么能力串联，是否值得合并。
- 再给产出物对照：用户多给什么，通常多拿到什么。
- 再给改善点：依赖、路径、破坏性、验证缺口。
- 最后给适合与边界：它能帮谁、不能做什么、文件之间怎么配合。

不要为了版式牺牲重点判断。除非用户明确说不要 HTML，否则完整解读都要生成 HTML 文件，并在对话里给出摘要和路径。

## HTML 报告

HTML 是默认交付物，用于存档、展示、分享或对比多个 skill。以下情况更要生成：

- 用户要保留一份可反复查看的 skill 说明书。
- 输入是多 skill 仓库，清单、关系和评分较复杂。
- 用户正在做汇报、交付或沉淀资料。

HTML 报告要做到：

- 自包含 HTML/CSS，不依赖远程字体或 CDN。
- 报告顶部必须有覆盖全部章节的可点击锚点目录（`.toc`），支持快速跳转到每一节。
- 优先使用 `assets/report-template.html`。
- 如果用户要求“高大上、设计感、image2、视觉母版”，先生成或参考视觉方向图，再把风格转译成可编辑 HTML/CSS；不要把整张图片当报告背景。
- 报告必须保留核心判断，不要只追求好看。必须包含：结论、快速判断、直接开始用提示词、搭配建议、产出物对照、完整度与改善点、适合谁用与不能做什么。
- 如果是包或多 skill 仓库，要包含清单表和工作流图。
- 报告文件保存在当前工作区；聊天回复给出摘要和 HTML 路径。
- 交付前必须按“交付前最小自检”检查结构、复制块、标签、文件关系图和旧章节残留。

使用模板时替换这些占位符：

- `{{REPORT_TITLE}}`
- `{{SKILL_NAME}}`
- `{{SUMMARY}}`
- `{{OBJECT_TYPE}}`：按需展示；不要为了填模板强行展示内部对象类型。
- `{{VERDICT}}`：当前判断，例如“可以使用”“先试用”“暂不建议安装”。
- `{{PAIRING_MODE}}`：按需展示；没有必要时不要强行展示。
- `{{SCORE}}`
- `{{TOKEN_ESTIMATE}}`：本次解读的预计 token，例如“中等 ~6-10k”。
- `{{SCORE_NOTE}}`
- `{{ONE_LINE_RECOMMENDATION}}`
- `{{OBJECT_CARDS}}`：快速判断格，默认放 skill 完整度、推荐动作、最大风险、直接使用风险，再加一个“token 消耗预估”格（共 5 格）。每格结构应是“label + 标色主判断 + 补充说明”，不要把关键值混在普通正文里；可用 `.ok`、`.hot`、`.warn` 或等效样式。token 消耗格放在最后，加 `class="span-all"` 让它横跨整行，容纳“分档 + 区间 + 省 token 建议”。
- `{{PREP_CARDS}}`：直接开始用下的补充信息，说明想让结果更准时可以提供什么。`Pro 版` / `Plus 版` 如果提供可复制提示词，必须使用 `.copy-block`，把复制按钮放在白色提示词框右下角。
- `{{FOLLOWUP_CARDS}}`：直接开始用底部的“失败了怎么反馈”。只保留最短反馈模板或反馈材料清单。
- `{{REAL_IO}}`：产出物对照，包括输入/输出对照、天花板示例、真实输入输出、产物、文件、脚本、API、环境变量、输出路径。
- `{{RISK_CARDS}}`：完整度、改善点、用户可感知影响、处理方式、修复咨询提示词；严重问题才标成风险。
- `{{PAIRING_STEPS}}`：单独用够不够、最佳搭配、后续搭配；只有发现相似 skill 时才展示合并建议。
  - 每个搭配对象都要把 `.badge` 状态标签放在 skill 名前面。
- `{{USAGE_CARDS}}`：适合谁用，不能做什么。包含“它能帮你做什么 / 什么时候最适合 / 不能做什么 / 文件关系图”等信息块。文件关系图先展示“包里有什么”，再展示“主流程：从输入到报告”，最后展示“旁路文件”。优先用上下排列的 `.relation-flow` / `.relation-step`，文件夹总览可以用 `.package-map` / `.package-cell`。
- `{{TRIAL_PROMPT}}`
- `{{FOOTER}}`

如果某一栏没有内容，改写结构，不要留下空占位符。

## 修复策略

解读阶段：

- 不默认修复。
- 发现问题先说明是否阻塞第一次试用。
- 给出建议动作：先试用、修复副本、安装、归档、深读另一个 skill。

修复阶段，只有用户明确要求时执行：

- 优先创建修复副本，不直接改原 skill。
- 保留原结构和路径，除非问题来自结构本身。
- 明确列出改了什么、为什么改、怎么验证。
- 如果用户还没试用过，优先建议最小修复或先做试用，不做大合并。
